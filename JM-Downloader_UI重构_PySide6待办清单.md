# JM-Downloader UI 重构待办清单

> 目标：将现有 `pywebview + Flask + HTML/CSS/JS` 桌面界面迁移为
> `PySide6 + Qt Widgets` 原生桌面界面，同时保留已经稳定的下载、PDF 和本地库能力。

## 一、范围与原则

本次重构只处理 UI 技术栈迁移和现有功能对等，不同时重写下载架构，也不把新产品功能混入迁移过程。

### Qt 迁移主线

- [x] 保留现有 `TaskManager`、`DownloadWorker`、`LibraryService` 和 PDF 逻辑。
- [x] 先建立 UI 无关的数据契约，再接入 Qt。
- [x] 下载、库扫描、PDF 重建等耗时操作不得阻塞 Qt 主线程。
- [x] Qt 版达到现有功能对等后，删除 Flask、pywebview、SSE 和 Web 静态资源。
- [x] 每个完整阶段都运行自动化测试，并尽早进行 PyInstaller 打包验证。
- [ ] 最终继续采用 Windows x64、PyInstaller onedir、文件夹便携发行。

### 不进入本次迁移的内容

以下内容独立列入后续版本，不作为移除 WebView2 的前置条件：

- 搜索、分类和排行榜。
- 漫画详情和章节选择。
- 元数据数据库和按标题、作者、标签搜索本地库。
- 全面 asyncio 架构。
- 异步、同步和 fallback 三套 Gateway。
- 下载恢复、缺页扫描和增量更新。
- 账号、收藏夹、订阅和自动更新。

### 当前技术约定

- 任务进度统一使用 `0 ~ 100` 整数，直接匹配 `QProgressBar` 和现有后端。
- 第一版只使用当前真实存在的任务状态：`pending`、`fetching`、`downloading`、`completed`、`failed`。
- 新任务状态必须先在业务层定义状态转换和测试，不能只在 UI 中增加。
- UI 不接收 `/api/...` 地址；任务和本地库向 Qt 提供本地 `Path` 或不可变快照。
- Qt 控制器直接消费业务快照，不在 UI 与业务层之间增加传输协议。
- 项目代码继续放在 `jm_downloader` 命名空间内，不新增顶层 `core`、`services`、`runtime` 等通用包。
- 配置和日志固定在可写的程序目录；图片和 PDF 默认存放在程序目录，用户可以明确选择外部目录。
- 便携验收要求默认配置不把持久业务数据写到程序目录之外；用户主动选择的外部输出目录不计为违规，也不要求阻止 Windows 自身的临时访问和最近文件记录。

## 二、目标结构

第六阶段完成后的正式结构：

```text
JM-Downloader/
├── jm_downloader/
│   ├── models.py              # UI 无关快照和状态
│   ├── tasks.py               # 任务队列
│   ├── downloader.py          # 下载工作线程
│   ├── library.py             # 本地库
│   ├── pdf.py
│   ├── settings.py
│   └── qt/
│       ├── app.py
│       ├── main_window.py
│       ├── controllers/
│       ├── pages/
│       │   ├── download_page.py
│       │   ├── library_page.py
│       │   └── settings_page.py
│       ├── widgets/
│       └── resources/
├── desktop.py                 # Qt 正式入口
├── JM-Downloader.spec
└── tests/
```

Qt 数据流：

```text
Qt Widget
   ↓ command
Qt Controller / Bridge
   ↓
TaskManager / LibraryService
   ↓ immutable snapshot / Qt Signal
Qt main thread
```

不在 Qt 与业务层之间增加本地 HTTP 服务。

## 十一、后续产品功能路线

这些内容在 Qt 正式版发布后分别规划，不阻塞 UI 迁移。

### 路线 A：详情和章节下载

- 漫画详情。
- 章节列表。
- 整本、单章和勾选章节下载。
- 已下载章节识别。
- 缺失章节补全。

### 路线 B：搜索与发现

- 关键词和标签搜索。
- 分页和可靠字段展示。
- 搜索结果进入详情或下载队列。
- 快速连续搜索的取消和结果顺序保护。

### 路线 C：本地库元数据

- 选择 JSON 清单或 SQLite 索引。
- 标题、作者、标签和下载时间持久化。
- 按标题、作者和标签搜索。
- 索引损坏恢复和旧目录迁移。

### 路线 D：下载可靠性

- 真正的协作式取消。
- 下载完整性扫描。
- 失败章节和失败图片列表。
- 增量下载和缺页修复。

### 路线 E：异步架构评估

只有满足以下条件之一才进入设计：

- 实际性能测试证明同步后台线程成为瓶颈。
- 上游异步 API 在目标版本稳定且有明确收益。
- 搜索、账号或长期后台任务需要统一事件循环。

即使进入评估，也应先做小型验证，不默认同时实现异步、同步和 fallback 三套运行路径。
